## Промпт (копировать целиком)

```
Я развиваю открытый исследовательский проект — методологию улучшения системного
промпта (CLAUDE.md) для AI-агента, пишущего код (Claude Code). Подход: любое новое
правило в системном промпте проверяется экспериментом (A: голая модель, B: текущий
промпт, C: промпт + кандидат) на синтетических задачах-ловушках, и внедряется
только если проходит строгую планку — минимум 10 независимых задач разных типов,
полная регрессионная проверка, и явный расчёт выгода/цена: (частота класса ошибки
в реальном коде × тяжесть последствий) против (постоянный токен-налог правила на
КАЖДУЮ будущую сессию агента, включая те, где правило не пригодится).

Проблема: у меня нет реальных количественных данных о частоте и тяжести классов
ошибок — только синтетические лабораторные эксперименты. Собери эмпирические
данные (научные статьи, индустриальные отчёты, результаты статического анализа
open-source кода, посты с количественными измерениями) по следующим темам.
Для КАЖДОГО найденного факта указывай источник (ссылка), тип источника
(рецензируемая статья / отчёт компании / блог с методологией / мнение без данных)
и, если есть, конкретную цифру (%, частота, medium/mean). Отдельно отметь, где
данных просто нет или они противоречивы — это тоже полезный результат.

## Блок 1. Частота и тяжесть классов ошибок в реальном коде

1. Пустые/подавленные catch-блоки и проигнорированные promise rejections: как
   часто встречаются в open-source кодовых базах (данные статического анализа,
   SonarQube/CodeQL/аналогичных инструментов), какая доля реальных инцидентов
   (post-mortems, CVE) связана именно с подавленной/потерянной ошибкой.
2. Дублированный (скопированный) код с багами: исследования clone detection —
   какая доля клонов кода содержит один и тот же баг, скопированный вместе с
   логикой; как часто фикс применяется только к одной копии, а баг остаётся в
   других (bug propagation через code clones).
3. Устаревшая документация/комментарии относительно поведения кода (doc drift):
   есть ли измерения, как часто комментарии/README расходятся с реальным
   поведением после рефакторинга, и есть ли данные о влиянии этого на
   продуктивность разработчиков или AI-агентов, использующих код как контекст.
4. Раздувание зависимостей: данные о стоимости добавления тривиальных npm/pip
   пакетов (левпад-класса) — влияние на security surface (supply chain атаки
   через транзитивные зависимости), размер бандла, maintenance burden;
   исследования о доле зависимостей, дублирующих возможности уже используемых
   библиотек или стандартной библиотеки языка.
5. Регрессии из-за неполного прогона тестов: данные о доле CI-инцидентов,
   вызванных тем, что изменение не запустило/не затронуло существующий тест
   другого модуля.

## Блок 2. Поведение LLM-агентов при автономном кодировании

6. Reward hacking / test-gaming в agentic coding: свежие статьи (2025-2026) о
   том, как часто модели «читерят» — ослабляют/удаляют тесты вместо починки
   кода, подгоняют решение под конкретные проверки вместо общего алгоритма.
   Есть ли количественные оценки частоты на бенчмарках типа SWE-bench.
7. Систематические провалы LLM при автономном рефакторинге/фиксе багов: ищи
   именно данные о том, ищет ли модель по умолчанию похожие места в кодовой
   базе после локального фикса (аналог нашей гипотезы dedup-grep-principle),
   обновляет ли документацию при смене поведения, добавляет ли зависимости
   без необходимости — если есть готовые таксономии провалов из бенчмарков
   (SWE-bench, SWE-bench Verified, LiveCodeBench, любые agentic coding evals),
   перечисли их категории ошибок.
8. Есть ли опубликованные количественные данные (не блог-посты без цифр) о
   влиянии system-prompt-файлов (CLAUDE.md/AGENTS.md-стиль) на качество кода
   агента — сравнение «с файлом правил» vs «без», любые метрики.

## Блок 3. Методология оценки (валидация нашего подхода)

9. Есть ли рекомендации/исследования о выборе модели для бенчмаркинга
   изменений system-prompt: имеет ли смысл гипотеза «оценивать на средней/
   слабой модели, чтобы не упереться в потолок способностей», или есть
   контраргументы (например, что слабая модель ведёт себя качественно иначе
   и выводы не переносятся на сильную)?
10. Best practices для A/B-тестирования system-prompt/agent-instructions:
    сколько повторов нужно для статистической значимости при известной
    вариативности LLM-вывода; как индустрия (Anthropic, OpenAI, авторы
    agent-фреймворков) тестирует изменения промптов перед релизом — есть ли
    открытые описания процесса (prompt regression testing, eval suites).

## Формат ответа

Структурируй ответ по блокам 1-3 (можно по под-пунктам 1-10). По каждому пункту:
факт → источник со ссылкой → тип источника → конкретная цифра, если есть. В конце
— отдельный список пунктов, где ты НЕ нашёл значимых данных (честное признание
пробела не менее ценно, чем факт). Пиши на русском, ссылки и цитаты можно
оставлять на языке оригинала.
```

## Что делать с результатом

1. Сохранить ответ Gemini как `research/findings/2026-07-08-gemini-deep-research-result.md` (дословно, с источниками).
2. Для каждого кандидата в `bench/candidates/README.md` — добавить строку в колонку «Источник/обоснование» со ссылкой на конкретный найденный факт (или явно отметить «данных не нашлось — гипотеза остаётся умозрительной»).
3. Если найдутся новые, подкреплённые данными классы ошибок — оформить как гипотезы по шаблону из `bench/candidates/README.md` («Как добавить новую гипотезу»), с пометкой источника, а не «мне показалось».
4. Использовать найденные частотные данные для расчёта «выгода/цена» (EXPERIMENTS.md §6, пункт 3) в карточках экспериментов — это единственный пункт планки внедрения, который синтетический стенд принципиально не может закрыть сам.
