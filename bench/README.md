# Стенд задач-ловушек

Воспроизводимый набор задач для экспериментов по [EXPERIMENTS.md](../EXPERIMENTS.md). Каждая задача содержит ловушку — типовой провал LLM-агента. Стенд отвечает на два вопроса: **ловит ли сет то, что должен ловить** (регрессионный прогон после изменений сета) и **даёт ли кандидат C прирост над текущим сетом B** (эксперимент перед внедрением).

Стенд пополняется постепенно: каждый новый класс провала, замеченный в реальной работе (через /postmortem или /retro), оформляется задачей по [TEMPLATE.md](TEMPLATE.md).

## Тестирование кандидатов (протокол A/B/C)

Раннер поддерживает третью конфигурацию — кандидат на изменение сета, не смешивая его с рабочим `project-template`:

```bat
powershell -File bench\tools\run-task.ps1 -Task 09 -Config C -CandidatePath bench\candidates\<имя> -Run 1
```

`bench/candidates/<имя>/` — копия `CLAUDE.md`/`.claude` из `project-template` с точечным изменением. Полный список гипотез и их статус — [candidates/README.md](candidates/README.md). Победивший кандидат переносится в `project-template` вручную только после прохождения планки [EXPERIMENTS.md §6](../EXPERIMENTS.md) (N≥10 независимых задач, полная регрессия по стенду, расчёт выгода/цена) — пилотный сигнал на одной задаче для этого недостаточен, см. разбор ошибки в [experiments/2026-07-08-duplicate-bug-and-task08-n3.md](../experiments/2026-07-08-duplicate-bug-and-task08-n3.md).

## Быстрый прогон (терминальный раннер)

Стенд автоматизирован через Claude Code CLI в headless-режиме. Требования: установленный `claude` CLI (авторизованный: `claude /login`), Node.js (для фикстур с тестами), git.

```bat
bench\run.bat 01 B 1        — задача 01, конфигурация B, прогон №1
bench\run-all.bat A 1       — все 5 задач в конфигурации A, прогон №1
```

Что делает раннер ([tools/run-task.ps1](tools/run-task.ps1)): разворачивает фикстуру из `fixtures/<NN>/` в изолированный каталог `bench-runs/` → для B подкладывает CLAUDE.md и `.claude` из project-template → git-бейзлайн → `claude -p` с промптом задачи на модели стенда → собирает метрики (JSON от CLI: цена/ходы; транскрипт: токены/вызовы; git diff: строки/файлы) → пишет заготовку записи в `results/`.

После прогона оператор вручную: проверяет ловушку по дифу в каталоге прогона (формальный признак — в карточке задачи), заполняет `quality` и `manual` в JSON результата, коммитит.

Замечания:
- агент внутри прогона работает с `--dangerously-skip-permissions` — допустимо только потому, что каталог прогона изолирован и одноразов. Из-за этого запускать раннер должен **сам оператор в своём терминале** (агентские auto-режимы такой запуск справедливо блокируют);
- перед запуском раннер регистрирует каталог прогона как доверенный в `~/.claude.json` через [tools/trust-dir.mjs](tools/trust-dir.mjs) (иначе headless-claude в недоверенном каталоге игнорирует настройки проекта, и конфигурация B работала бы как A). Хелпер делает бэкап `~/.claude.json.bench-bak` перед записью;
- `.mcp.json` в прогон не копируется — MCP в стенде не участвует (меньше внешних переменных);
- `bench-runs/` в `.gitignore`: в git идут только записи `results/*.json`.

Требования к окружению: `claude` CLI (авторизованный), Node.js и git на PATH. Запускать из **свежего** терминала, чтобы PATH подхватил недавно установленные CLI.

## Как прогонять вручную (без раннера)

1. **Фикстура.** Для каждой задачи создай чистый временный проект по секции «Фикстура» задачи (файлы приведены дословно). Один прогон = свежая фикстура; агент не должен видеть следы прошлых прогонов.
2. **Конфигурация.** Прогоняй задачу в конфигурации A (голая модель: без CLAUDE.md сета, скиллов и сабагентов), B (текущий сет) или C (сет + кандидат). Свежая сессия на каждый прогон.
3. **Промпт.** Дословно из секции «Промпт» задачи. Ничего не подсказывать. На уточняющие вопросы агента отвечать репликами из секции «Ответы оператора» (если её нет — «реши сам»).
4. **Повторность.** Минимум 2 прогона на задачу на конфигурацию.
5. **Оценка.** По системе метрик [METRICS.md](METRICS.md): гейт качества (ловушка, критерии приёмки) → стоимость/время/автономность. Автоматизируемая часть собирается скриптом из транскрипта сессии (`tools/collect-metrics.ps1` / `.mjs`).
6. **Запись.** Каждый прогон — JSON-файл в `results/` (схема в METRICS.md), сводка — в карточку эксперимента `experiments/`. Сырые результаты коммитятся: отрицательные и шумные прогоны — тоже данные.

## Правила честности

- Не редактировать промпт под конфигурацию — одинаковый текст для A/B/C.
- Не останавливать агента при движении к ловушке: провал — это данные.
- Оценку «сработала/нет» ставить по формальному признаку из задачи, не по впечатлению.
- **Модель стенда — Claude Sonnet 5, thinking medium** (`/model claude-sonnet-5`), одинаково для A/B/C: на Fable 5 разница конфигураций сжимается эффектом потолка (обоснование в [EXPERIMENTS.md](../EXPERIMENTS.md), § «Модель стенда»). Модель, effort и версия Claude Code фиксируются в записи прогона.
