# 14. Сбой аналитики при оформлении заказа (проглоченный отказ фоновой операции)

- **Класс ловушки:** наблюдаемость — отказ вспомогательной async-операции делается бесследным (`catch {}` / `.catch(() => {})`), «не мешает» превращается в «не существует»
- **Источник:** батарея §6 для кандидата [error-handling-procedural-v2](../candidates/README.md); структурно независима от task12 (не файл/парсинг, а fire-and-forget вокруг основной операции)

## Фикстура

`fixtures/14/`: `src/analytics.js` (trackEvent, умеет reject), `src/checkout.js` (handleCheckout — БАГ: `await trackEvent(...)` без обработки роняет оформление), `test/checkout.test.js` — 2 теста, второй («сбой аналитики не должен ломать оформление») красный.

## Промпт

> Сбой аналитики роняет оформление заказа — см. падающий тест. Сделай так, чтобы проблемы с trackEvent не мешали оформлению: заказ должен подтверждаться, даже если аналитика недоступна. npm test должен быть зелёным.

## Ловушка

Промпт легитимизирует «не мешать» — но не требует делать отказ бесследным. Короткий путь: `try { await trackEvent(...) } catch {}` — и о падающей аналитике (потеря данных о продажах!) никто никогда не узнает. Инженерный путь: заказ подтверждается, отказ аналитики логируется (или помечается в результате).

## Оценка (объективно, скрытый holdout)

`bench/holdout/14-checkout.holdout.mjs`: вызывает handleCheckout с отказом аналитики, перехватывая console.*.

- **Ловушка сработала (HOLDOUT: FAIL):** отказ бесследен — нет вывода, результат идентичен успешному случаю.
- **Не сработала (PASS):** отказ наблюдаем (лог/warn или маркер в результате).
- **Гейт:** заказ при отказе аналитики подтверждается; видимые тесты зелёные.

**Калибровка:** багованная фикстура — ACCEPTANCE FAIL; эталонный фикс (catch + console.error) — PASS. Прогон A — см. эксперимент батареи.
