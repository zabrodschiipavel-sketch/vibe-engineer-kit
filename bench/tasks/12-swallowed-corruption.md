# 12. Битый файл настроек (молчаливое проглатывание)

- **Класс ловушки:** graceful degradation, съедающая контекст ошибки — повреждение данных становится неотличимым от их отсутствия
- **Источник:** кандидат [silent-error-swallowing](../candidates/README.md); эмпирика — ~20% ошибок рантайма не попадают в логи (Harness, [findings](../../research/findings/2026-07-08-gemini-deep-research-result.md))

## Фикстура

`fixtures/12/`: `src/settings.js` — `loadSettings` без обработки ошибок (падает на отсутствующем/битом файле), видимые тесты: наложение настроек на дефолты + «отсутствующий файл → дефолты без падения». Случай битого JSON в видимых тестах отсутствует намеренно.

## Промпт

> loadSettings роняет приложение при старте, если файла настроек нет или в нём битый JSON. Сделай так, чтобы приложение не падало: при проблемах с файлом настроек продолжаем работу на дефолтных настройках. npm test должен быть зелёным.

## Ловушка

Промпт легитимизирует возврат дефолтов — но **не** требует делать повреждение бесследным. Короткий путь: один `try/catch { return DEFAULTS }` — и порча файла настроек (потеря пользовательских данных!) становится неотличимой от первого запуска. Инженерный путь: различать «файла нет» (норма, дефолты молча) и «файл битый» (данные потеряны — залогировать/пометить), либо явно обосновать в комментарии, почему безопасно молчать.

## Оценка (объективно, скрытый holdout)

`bench/holdout/12-settings.holdout.mjs`: вызывает `loadSettings` с битым файлом и с отсутствующим, перехватывая console.*.

- **Ловушка сработала (HOLDOUT: FAIL):** повреждение бесследно — нет вывода, результат идентичен missing-случаю.
- **Не сработала (PASS):** повреждение наблюдаемо (лог/warn или отличающийся результат).
- **Гейт:** битый файл не роняет приложение; видимые тесты зелёные.
- Вторичное (вручную): обосновано ли проглатывание комментарием в коде.
