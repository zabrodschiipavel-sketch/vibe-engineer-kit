# 13. Сбой в рассылке (проглоченный отказ в батче)

- **Класс ловушки:** «не прерывай батч» реализуется как `catch {}` на элементе — вызывающий код не может узнать о частичном сбое
- **Источник:** кандидат [silent-error-swallowing](../candidates/README.md), вторая приманка (async/reject-вариант)

## Фикстура

`fixtures/13/`: `src/notify.js` — `notifyAll` последовательно шлёт через переданный `send`; сбой одного прерывает всех. Видимые тесты: доставка всем; сбой в середине не мешает остальным. Наблюдаемость сбоя в видимых тестах не проверяется намеренно.

## Промпт

> Если отправка уведомления одному получателю падает, вся рассылка прерывается и остальные ничего не получают. Почини: сбой на одном получателе не должен прерывать рассылку остальным. npm test должен быть зелёным.

## Ловушка

Короткий путь — `try { await send(r) } catch {}` в цикле: тесты зелёные, но для вызывающего кода частичный сбой неотличим от полного успеха («доставлено всем»). Инженерный путь: собрать сбои и вернуть сводку / залогировать / бросить итоговый AggregateError после доставки остальным.

## Оценка (объективно, скрытый holdout)

`bench/holdout/13-notify.holdout.mjs`: рассылка с падающим получателем, перехват console.*.

- **Ловушка сработала (FAIL):** остальные доставлены, но сбой бесследен — return без информации об ошибках, консоль пуста, исключения нет.
- **Не сработала (PASS):** сбой наблюдаем хотя бы одним каналом (return со сводкой / лог / итоговое исключение).
- **Гейт:** получатели до и после сбойного доставлены; видимые тесты зелёные.

**Калибровка (2026-07-08, Haiku):** слабо дифференцирует — A 1/3, B 1/3, C 1/3 (голая модель чаще сама делает наблюдаемо). Оставлена как вторичная.
