# /investigate — сначала пойми, потом чини

Твоя задача — найти и **доказать** корневую причину. Правка кода до доказанной причины запрещена: фикс «наугад» либо маскирует симптом, либо ломает что-то ещё. Антипаттерн вайбкодера «перегенерируй, вдруг заработает» — ровно то, что этот скилл искореняет.

## Шаг 1. Воспроизведение

- Добейся минимального воспроизведения: наименьший вход/сценарий, на котором баг проявляется стабильно.
- Не воспроизводится? Не чини вслепую — собирай данные: логи, точный вход, окружение, частота. «Иногда падает» — это отсутствие данных, а не описание.

## Шаг 2. Факты

- Точный симптом: что происходит vs что ожидалось (дословно, с текстом ошибки).
- Когда началось: `git log` по затронутой области; если регрессия — `git bisect` (воспроизводящий скрипт из шага 1 = идеальный тест для bisect).
- Граница: на каких входах/конфигурациях работает, на каких нет. Разница между «работает» и «не работает» — самый информативный сигнал.

## Шаг 3. Гипотезы

- Выпиши список гипотез, к каждой — **дешёвую проверку**, которая её подтвердит или убьёт.
- Проверяй по одной, фиксируй результат каждой (подтверждена/опровергнута/не проверяема). Не бросай гипотезу на полпути ради новой блестящей.
- Читай код и ошибку сам, а не «по вайбу»: стектрейс — это карта, а не заклинание для копипасты.

## Шаг 4. Доказательство

Корневая причина доказана, когда ты можешь показать **механизм**: такой-то вход → такой-то путь в коде → вот здесь состояние становится неверным → симптом. «Похоже на race condition» — не причина, а гипотеза. Проверь механизм направленным экспериментом (логирование в подозреваемой точке, изолированный вызов).

## Шаг 5. Отчёт и передача в фикс

Выдай: симптом → корневая причина → доказательство → предлагаемый фикс → что ещё может быть затронуто той же причиной (Grep по аналогичным местам).

Фикс начинается с воспроизводящего теста (красный), затем правка (зелёный). «Починилось само» = не починилось: причина не найдена, баг вернётся. После фикса — `/postmortem`, чтобы причина стала правилом.
