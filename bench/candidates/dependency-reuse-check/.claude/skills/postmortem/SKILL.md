# /postmortem — баг → правило

Цель: каждый нетривиальный баг конвертируется в строчку конфига, после которой агент (и человек) физически не сможет повторить ошибку. Без обвинений — только механика.

## Шаг 1. Факты

Собери из контекста/гита/вопросов пользователю:
- Что сломалось (симптом) и как заметили?
- Когда внесено (найди коммит: `git log`, `git blame` по затронутым строкам)?
- Как починили?

## Шаг 2. 5 «почему»

От симптома копай вниз, пока не упрёшься в **процессную** причину, а не персональную:

> Упал прод → почему? Null в ответе API → почему не отловили? Нет теста на пустой ответ → почему? Спека не описывала граничный случай → почему? /spec не спрашивал про поведение при недоступном API.

«Разработчик ошибся» — не корневая причина. Корневая причина — то, что позволило ошибке дойти до прода.

## Шаг 3. Правило

Сформулируй правило, которое отсекло бы этот баг **автоматически или на ревью**. Хорошее правило:
- конкретное («все клиенты внешних API обязаны иметь тест на пустой/ошибочный ответ»), а не благое пожелание («быть внимательнее»);
- проверяемое — по нему можно спросить «выполнено или нет».

## Шаг 4. Записать

1. Добавь правило в `CLAUDE.md`, секция **«Выученные правила»**, с датой и однострочной ссылкой на инцидент.
2. Если правило можно закрепить жёстче — предложи: тест, линт-правило, hook, запись в `permissions.deny`.
3. Если баг указал на дыру в процессе (например, /spec не спрашивает про граничные случаи X) — предложи правку соответствующего скилла.

## Шаг 5. Мини-отчёт

Выдай пользователю: симптом → корневая причина → правило → куда записано. Пять строк, без воды.
