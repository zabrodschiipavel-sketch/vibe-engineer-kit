# Кандидаты и гипотезы

Индекс всех гипотез об улучшении сета. Планка внедрения в рабочий `project-template/CLAUDE.md` — [EXPERIMENTS.md §6](../../EXPERIMENTS.md): N≥10 независимых задач, полный регрессионный прогон по стенду, расчёт выгода/цена, проверка на обобщение и на дублирование с уже существующими принципами. ## Наблюдение по итогам первых пилотов (N=2, гипотеза, не закон)

Успешный пилот (dedup-grep-principle, эффект 3/3) и инертный (silent-error-swallowing, эффект 0/9) различаются формой правила: **процедурное** («после починки бага — сделай grep похожих мест») сработало, **нормативное** («не глотай ошибки молча») — перекрылось рамкой задачи, легитимизировавшей поведение. При написании новых кандидатов предпочитать процедурную форму: конкретное действие, привязанное к конкретному триггеру.

**Уточнение по [эксперименту изоляции 2026-07-09](../../experiments/2026-07-09-core-isolation.md):** нормативное правило не бесполезно — оно инертно *в одиночку* (CLAUDE.md «спроси при неоднозначности» без скиллов: 0 эффекта на task03), но в связке с процедурным крючком (/spec, /ship) даёт надёжность, которой нет у крючка поодиночке (авто-триггер /spec без CLAUDE.md: 1/3). Рабочая формула: **норма в CLAUDE.md + процедурный скилл, на который она опирается**.

**Чек-лист формулировки правила по [линии error-handling 2026-07-09](../../experiments/2026-07-09-error-handling-procedural-pilot.md)** (три итерации одного правила: 0/9 вовлечённости → 3/3 вовлечённости при 1/3 исхода → 3/3 исхода): (а) триггер привязан к действию агента («добавил catch»), не к свойству кода; (б) **все** разрешённые ветки ведут к цели правила — иначе агент выберет самую дешёвую (под-паттерн «дешёвая ветка»: комментарий-обоснование выродился в самооправдание); (в) формулировка явно снимает мнимый конфликт с рамкой задачи («не падать» ≠ «молчать»).

**Третья итерация формулы по [зонду плотности 2026-07-09](../../experiments/2026-07-09-density-probe-pilot.md)** (лестница из 6 конфигураций): механизмов два. Описания скиллов/агентов в постоянном контексте создают *процессный приор* — его хватает для дисциплинарных побед (task06: 0/3 при нейтрализованных телах). Тела-процедуры нужны для *контр-рефлексных* побед, где требуемое поведение противоречит дефолту «сделать задачу немедленно» (task03: без тела /spec остановка ненадёжна, 2/3 ловушек). При написании кандидатов: сначала спроси, борется ли правило с рефлексом агента — если да, нужна процедура с триггером, если нет — может хватить описания/нормы.

## Как добавить новую гипотезу

1. Сформулируй проверяемо: «принцип X снизит долю провалов класса Y на задачах Z» (не «X — это хорошая практика»).
2. Проверь на дублирование с уже существующими пунктами CLAUDE.md (см. таблицу выше — не первый случай, когда идея оказывается уже покрыта).
3. Если не дублирует — скопируй `project-template` в `bench/candidates/<slug>/`, добавь ОДНУ формулировку с пометкой `[КАНДИДАТ, не пилотирован]`.
4. Добавь строку в таблицу «Активные кандидаты» этого файла.
5. Пилот — по протоколу EXPERIMENTS.md §1-5 (сначала нужна фикстура-ловушка в `bench/fixtures/`, лучше со скрытым holdout). Пилотный сигнал переводит статус в 🔬⏳, не в ✅.
6. Внедрение в `project-template` — только после прохождения §6 целиком.
