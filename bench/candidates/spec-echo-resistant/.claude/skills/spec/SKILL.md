# /spec — спецификация до кода

Ты получаешь «хотелку» — неформальное описание желаемого. Твоя задача — превратить её в спецификацию, по которой можно строить план и код. **Не пиши ни строчки кода на этом этапе.**

## Шаг 1. Задай вопросы — первым же сообщением

Сначала изучи релевантную часть кодовой базы (что уже есть, что можно переиспользовать). Затем задай пользователю уточняющие вопросы — минимум те, где ответ меняет решение:

- Кто пользователь этой фичи и какой сценарий главный?
- Что явно НЕ входит в задачу?
- Какие граничные случаи важны (пустые данные, конкуренция, ошибки внешних систем)?
- Есть ли ограничения: совместимость, производительность, дедлайн-компромиссы?

Как задавать: есть инструмент AskUserQuestion — используй его; нет — задай те же вопросы обычным текстом, нумерованным списком, и заверши ход. Оба пути равнозначны: цель шага — чтобы пользователь получил конкретные вопросы, на которые можно ответить.

Результат этого шага — **сами вопросы в твоём сообщении**, не намерение их задать. Перед отправкой проверь два признака: (1) в сообщении стоят конкретные вопросы про ЭТУ задачу; (2) в нём нет мета-фраз о процессе («сейчас сформирую вопросы», «ожидаю деталей», «выполнил /spec»). Сообщение, пересказывающее эти инструкции вместо вопросов, — провал шага, а не его выполнение.

Не додумывай ответы сам. Каждый неотвеченный вопрос станет галлюцинацией в коде.

## Шаг 2. Спецификация

После ответов создай файл `docs/specs/<дата>-<slug>.md` (или `specs/`, если в проекте так принято):

```markdown
# Спецификация: <название>

## Цель
Одним абзацем: какую проблему решаем и для кого.

## Границы
- Делаем: ...
- НЕ делаем: ...   ← эта часть важнее первой

## Решение
Как будет работать: потоки данных, изменения в API/схеме, затронутые модули.
Что переиспользуем из существующего кода (конкретные файлы/функции).

## Критерии приёмки
- [ ] Проверяемые утверждения. Каждое можно превратить в тест.

## План верификации
Как проверим end-to-end: команды, сценарии ручного QA, граничные случаи.

## Риски
Что может пойти не так, зоны повышенного внимания (auth/данные/миграции).
```

## Шаг 3. Декомпозиция на задачи

После одобрения спецификации разбей решение на нумерованные задачи (паттерн spec-kit: спека должна быть исполняемой). Добавь в тот же файл секцию:

```markdown
## Задачи

1. [ ] <атомарная задача — один проверяемый шаг> → проверка: <команда/тест/наблюдение>
2. [ ] ... → проверка: ...
```

Правила декомпозиции:
- каждая задача завершается компилирующимся, проходящим тесты состоянием;
- у каждой задачи есть **своя** проверка — «сделать X» без «как проверю» не задача, а пожелание;
- зависимости между задачами — явные (задача 3 после 1 и 2);
- задачи по рискованным зонам (auth/данные/миграции) помечай ⚠ — им построчное ревью.

## Шаг 4. Передача в работу

Покажи спецификацию с задачами пользователю. После одобрения предложи перейти в plan mode для построения плана реализации. Критерии приёмки из спеки — это будущие тесты (test-first: сначала падающий тест по критерию, потом реализация). По мере выполнения отмечай чекбоксы задач — файл спеки и есть трекер прогресса.
