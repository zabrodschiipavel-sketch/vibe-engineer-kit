# /vibe-check — не превращается ли репо в вайбкод?

Просканируй кодовую базу на симптомы вайбкодинга и выдай отчёт с оценкой. Только диагностика — ничего не исправляй без отдельной команды пользователя.

## Что искать

Используй Grep/Glob по исходникам (исключи `node_modules`, `dist`, `build`, vendored-код):

1. **TODO-кладбище** — `TODO|FIXME|HACK|XXX` без ссылки на тикет. Посчитай и покажи самые старые (git blame по нескольким).
2. **Мёртвый код** — закомментированные блоки кода (не документация), файлы/функции, на которые нет ссылок, неиспользуемые экспорты.
3. **Отладочный мусор** — `console.log|print(|dbg!|var_dump` вне логгера, в продакшн-путях.
4. **Хардкод** — захардкоженные URL, IP, ключи-заглушки, магические числа там, где просится конфиг; литеральные данные, которые должны приходить из API.
5. **Дублирование** — похожие функции/утилиты в разных местах (частый след агента, не нашедшего существующий хелпер).
6. **Ослабленные тесты** — `skip|only|todo` в тестах, пустые тест-тела, assertions вида «не упало — и ладно».
7. **Гиганты** — файлы/функции аномального размера относительно остального репо (компонент на 1000 строк — классика вайбкода).
8. **Подавленные ошибки** — пустые `catch`, `# type: ignore`/`@ts-ignore`/`eslint-disable` без комментария-обоснования.

## Отчёт

Сгруппируй по категориям, по каждой: количество, 3-5 худших примеров с путями (`file:line`), оценка серьёзности. В конце:

- **Вердикт**: чистый инженерный код / местами вайбит / вайбкод (с обоснованием);
- **Топ-5 действий** — что почистить в первую очередь, отсортировано по соотношению «польза/усилие»;
- Предложи вынести крупные находки в отдельные задачи, а не чинить всё разом.

Тон отчёта — прямой: цифры и примеры, без утешений.
