# Сет вайбинженера

> Вайбкодер — человек с кучей непонятного ему и неработающего кода.
> Вайбинженер — человек с процессом, скиллами, сабагентами и Fable 5, который отвечает за каждую строчку в своём репозитории.

---

## 1. Философия

Термин **vibe engineering** ввёл Саймон Уиллисон в октябре 2025 года — как противоположность vibe coding: «безответственному способу строить софт, полностью управляемому промптами, без внимания к тому, как код на самом деле работает». Вайб-инжиниринг — это когда опытный человек **ускоряется** с помощью LLM, оставаясь ответственным за результат.

Ключевой тезис Уиллисона: **AI усиливает существующий опыт**. Это инструмент, который делает сильных сильнее, а слабых — быстрее закапывает. Статистика это подтверждает:

- **36% вайбкодеров вообще не тестируют** свои проекты;
- **68%** описывают результат как «быстро, но с изъянами», ещё 19% — как «хрупкий»;
- по данным систематического обзора ICSE 2026, **QA — самый игнорируемый этап** AI-разработки.

### Инвариант вайбинженера

> **В репозитории нет ни одной строчки, которую я не могу объяснить.**

Всё остальное — следствия. LLM пишет код; человек владеет системой. Если ты не можешь объяснить строчку — ты не сэкономил время, ты взял кредит под грабительский процент.

## 2. Петля вайбинженера: Spec → Plan → Build → Verify → Review → Ship → Learn

Один полный оборот петли = одна фича/фикс. Не пропускай этапы — статистика выше состоит из людей, которые пропускали.

### 2.1 Spec — спецификация до кода

Команда: `/spec <хотелка>`

Самый большой сдвиг в AI-разработке — перенос усилий с реализации на **спецификацию**. Прежде чем агент напишет хоть строчку:

1. Агент **обязан задать уточняющие вопросы** (это встроено в скилл). Не отвечай «делай как лучше» — каждый неотвеченный вопрос станет галлюцинацией.
2. На выходе — документ: цель, границы («что НЕ делаем» важнее «что делаем»), критерии приёмки, план верификации.
3. Спецификация сохраняется в репозиторий — это контекст для будущих сессий.

### 2.2 Plan — план в plan mode

Включи **plan mode** (Shift+Tab в Claude Code). Паттерн deep-work session:

1. Разведка: сабагент **scout** (дешёвая модель, read-only) собирает карту релевантного кода — что уже есть, что можно переиспользовать.
2. Fable 5 строит план: какие файлы, в каком порядке, что проверяем на каждом шаге.
3. Ты читаешь план и **одобряешь или правишь**. Минута на чтение плана экономит час на разгребание.

Правило: чем крупнее изменение — тем обязательнее plan mode. Правка одной строки плана не требует; миграция — требует всегда.

### 2.3 Build — малыми проверяемыми шагами

- **Test-first, где возможно**: сначала падающий тест, потом реализация до зелёного. Падающий тест — лучшая постановка задачи для агента: и спецификация, и критерий приёмки в одном флаконе.
- Каждый шаг должен компилироваться и проходить тесты. Не давай агенту уехать на 500 строк без чекпоинта.
- **Частые мелкие коммиты** с осмысленными сообщениями. Git — твоя машина времени; вайбкодер откатывает руками, вайбинженер — `git revert`.
- Явно передавай контекст: конкретные файлы, существующие хелперы и утилиты для переиспользования. Главная причина плохого AI-кода — агенту не дали контекст, и он изобрёл свой велосипед рядом с твоим.

### 2.4 Verify — не верь «должно работать»

Команда: `/verify` (встроенный скилл Claude Code) или руками.

- Прогони фичу **end-to-end**: запусти приложение, пройди сценарий, посмотри глазами. «Тесты зелёные» ≠ «работает».
- Ручной QA граничных случаев: пустой ввод, огромный ввод, отмена на середине, повторный сабмит.
- Для UI — визуальная верификация через браузерный MCP (скриншот до/после).

### 2.5 Review — адверсариальное ревью

- `/code-review` + сабагент **critic**. Критик работает в свежем контексте — это принципиально: агент, который писал код, психологически «защищает» его; агент с чистым контекстом — нет.
- **Глубина ревью по риску**: auth, платежи, работа с данными, миграции — построчно; шаблонный CRUD — бегло. Ревьюить всё одинаково — значит ревьюить плохо.
- **Читай каждый дифф перед коммитом.** Без исключений. Это то самое место, где вайбкодинг превращается в вайб-инжиниринг.
- Проверь на **reward hacking**: агент мог «починить» падающий тест, ослабив сам тест. Диф тестов читается внимательнее дифа кода.

### 2.6 Ship — конвейер сдачи

Команда: `/ship`

Чек-лист: тесты → линт → чтение полного дифа → «могу объяснить каждую строку?» → проверка, что тесты не ослаблены → коммит с сообщением, объясняющим *зачем*, а не *что*.

### 2.7 Learn — постмортем вместо амнезии

Команда: `/postmortem <описание бага>`

После каждого нетривиального бага: 5 «почему» → корневая причина → **правило в CLAUDE.md** (секция «Выученные правила»). Вайбкодер наступает на грабли циклически; вайбинженер конвертирует каждый баг в строчку конфига, после которой агент физически не сможет повторить ошибку.

## 3. Гигиена контекста

Контекст — главный ресурс. Всё, что в нём лежит, влияет на качество; всё лишнее — деградирует его.

- **Одна задача = одна сессия.** Закончил фичу — новая сессия. Не таскай хвост из трёх прошлых задач.
- **CLAUDE.md — короткий и точный.** Паттерн AGENTS.md-driven development («убирает 80% жалоб „ИИ не понимает мой проект“»): контекст проекта, команды, конвенции, текущее состояние, известные проблемы. Обновляй **по ходу** работы, а не «потом» — «потом» не наступает.
- **Журнал решений.** Каждое архитектурное решение — строчка: что решили, почему, что отвергли. Через месяц агент (и ты) не будет заново изобретать отвергнутое.
- **Контекст распух — компактись.** Длинная сессия с мусором хуже новой сессии с выжимкой: попроси резюме состояния, начни заново.
- **Память — для незапоминаемого.** IP-адреса, ключи окружений, договорённости с командой. Не дублируй в память то, что и так лежит в репозитории — это создаёт рассинхрон.
- **Родной язык — это нормально.** Инструкции на русском часто точнее передают нюанс, чем твой английский. Точность контекста важнее языка.

## 4. Арсенал Fable 5

Fable 5 — не «автодополнение на стероидах», а исполнитель, которому делегируют. Управляй им как инженер:

| Инструмент | Когда включать |
|---|---|
| **Plan mode** | Любое изменение крупнее пары файлов. Разведка → план → одобрение. |
| **Сабагенты** | Разведка (scout, дёшево), независимое ревью (critic), тесты по спеке (test-smith). Каждый — в своём чистом контексте. |
| **Workflow / фан-аут** | Крупные миграции, аудиты, ревью по многим измерениям — десятки параллельных агентов с адверсариальной проверкой находок. |
| **Ротация моделей** | Механика и разведка — haiku; архитектура и сложный код — Fable. Экономия до ~70% без потери качества там, где качество нужно. |
| **Worktree-изоляция** | Параллельные правки в одном репо — каждому агенту свой worktree, никаких конфликтов. |
| **Effort-уровни** | Низкий — на рутину, высокий — на верификацию и архитектуру. |
| **Память** | Факты между сессиями (см. `global/memory-seed.md`). |

Отдельный навык — **интуиция делегирования**: что отдать агенту (шаблонный CRUD, тесты, миграции по образцу, рефакторинг с тестовой сеткой), а что делать руками или под построчным контролем (auth, платежи, криптография, нетривиальные алгоритмы, всё необратимое).

## 5. MCP и плагины: подключай под задачу, а не «все сразу»

Каждый подключённый MCP-сервер ест контекст своими описаниями инструментов. Принцип: **минимальный постоянный набор + подключаемое по задаче**.

### Постоянный набор (ядро)

| Что | Зачем |
|---|---|
| **Context7** (`.mcp.json` в комплекте) | Актуальные доки любой библиотеки. Лечит главную болезнь LLM — устаревшие API в обучающих данных. Работает без ключей. |
| **GitHub MCP / `gh` CLI** | PR, issues, CI-статусы, не выходя из сессии. |
| **Плагин engineering** | Готовые скиллы: code-review, debug, architecture (ADR), incident-response, testing-strategy. |

### Подключаемое по профилю задачи

| Что | Когда |
|---|---|
| **Браузерный MCP** (Claude in Chrome / Playwright / Preview) | Веб-разработка: визуальная верификация UI, e2e-прогоны, чтение консоли и network. |
| **Hugging Face MCP** | Данные/ML: поиск моделей, датасетов, статей. |
| **Sentry / Datadog MCP** | Прод-отладка: стектрейсы и метрики прямо в контекст. |
| **БД-MCP** (Postgres и т.п.) | Только read-only доступ, если вообще нужен. |

Правило гигиены: если MCP не пригодился за неделю — отключи.

## 6. Технологические предпочтения вайбинженера

Стек выбирается в том числе за «дружелюбность к агенту»:

- **Статическая типизация** (TypeScript, а не JS; типизированный Python). Компилятор — бесплатный ревьюер, который проверяет каждую строчку агента до тебя.
- **Популярные мейнстрим-решения.** LLM знает React лучше свежего нишевого фреймворка. Нестандартный выбор — только с веским основанием, зафиксированным в журнале решений.
- **Предсказуемая структура проекта.** Агент, как и новый сотрудник, продуктивнее там, где всё лежит на ожидаемых местах.
- **Линтер + форматтер + CI с первого коммита.** Автоматизация — это то, что позволяет агенту итерироваться безопасно.

## 7. Чек-лист самодиагностики: не превращаюсь ли я в вайбкодера?

Команда `/vibe-check` проверяет репозиторий; этот список — проверяет тебя:

- [ ] «Перегенерируй, вдруг заработает» — вместо чтения ошибки.
- [ ] Коммит без прогона и без чтения дифа.
- [ ] Копипаста стектрейса в чат, не прочитав его самому.
- [ ] Диф на 2000 строк одним куском — «потом посмотрю».
- [ ] Ворнинги? Какие ворнинги.
- [ ] Тест внезапно «починился» — а ты не проверил, не подогнал ли агент тест под ответ.
- [ ] В коде есть места, про которые ты думаешь «ну, оно работает, не трогаю».

Один пункт — звоночек. Три — ты вайбкодер с подпиской.

## 8. Как пользоваться комплектом

1. Скопируй содержимое `project-template/` в корень своего проекта.
2. Заполни `CLAUDE.md` под свой стек (команды build/test/lint — обязательно).
3. По желанию перенеси `skills/` и `agents/` в `~/.claude/` — тогда они будут доступны во всех проектах.
4. Работай по петле из раздела 2. Первую неделю — принудительно, дальше — на автомате.
