## Зачем

Планка §6 требует N≥10 независимых задач. У кандидата есть 2 (task12, 13 — родственные). Эта часть добавляет 3 структурно независимые ловушки того же класса «наблюдаемость сбоя»:

| # | Ловушка | Структурное отличие |
|---|---|---|
| [14](../bench/tasks/14-swallowed-async-failure.md) | бесследный отказ фоновой async-операции (аналитика при заказе) | fire-and-forget вокруг основной операции |
| [15](../bench/tasks/15-dead-worker-silence.md) | молчаливо деградирующий вечный воркер (поллер) | долгоживущий цикл, повторяющиеся сбои, канал может быть API |
| [16](../bench/tasks/16-partial-batch-loss.md) | молчаливая потеря части батча (конвертер) | чистая функция, несколько сбоев за вызов, канал может быть в return |

Все три: видимые тесты легитимизируют «не падать/не прерываться», наблюдаемость видимыми тестами не проверяется, скрытый holdout ловит бесследность. Калибровка «багованная фикстура FAIL / эталонный фикс PASS» пройдена при проектировании.

## Протокол (этапы, зафиксированы до прогонов)

1. **Калибровка на A** (урок dependency-reflex: сначала проверить, что голая модель вообще падает): A × N=3 на каждой задаче. Ловушка валидна для батареи, если A ≥ 2/3. Задача с A ≤ 1/3 помечается слабой (как task13) или отбраковывается.
2. **B × N=3** на валидных ловушках — проверить, что текущий сет (без правила) тоже падает (ожидание: как task12 — B=A).
3. **C-v2 × N=3** на валидных ловушках — проверяемое: правило переносится на новые структуры класса (ожидание: 0/3).
4. Свод в таблицу A/B/C-v2 по 5 задачам класса (12, 13, 14, 15, 16). Для §6 останется: регрессия v2 по task01–11 (не ломает ли правило остальной стенд) и честный расчёт выгода/цена.

4. **Обобщение** — формулировка без деталей фикстур, пять разных структур, пять разных правильных фиксов.
5. **Дублирование** — существующие пункты CLAUDE.md обработку ошибок не покрывают (проверено при оформлении кандидата).
6. **Бюджет файла** — +1 строка; компенсация: кандидат заменяет собой потребность в двух отклонённых формулировках.
