# Эксперимент: перенос сета на чужую семью моделей (Gemini 3.1 Flash-Lite), пилот A/B/C

- **Дата:** 2026-07-10
- **Пункт роадмапа:** EXPERIMENTS.md §2а (переносимость правил между моделями); открытый пункт наблюдения error-handling-v2; батарея dedup-grep-principle
- **Статус:** завершён (пилот, N=2 — сигнальный, не §6)

## Гипотеза

Выигрыши сета B и пилотный сигнал кандидата dedup-grep-principle, полученные на Claude Haiku 4.5, воспроизводятся на модели чужой семьи — Gemini 3.1 Flash-Lite (Google) — через прокси-шлюз Claude Code Router. Литература ([findings 2026-07-08](../research/findings/2026-07-08-gemini-deep-research-result.md)) предсказывает частичный перенос с ослаблением.

## Инфраструктура (новое)

- Claude Code CLI работает на Gemini через шлюз CCR (`127.0.0.1:3456`, трансляция Anthropic⇄Gemini API); модель `gemini-3.1-flash-lite` — максимальная бесплатная квота Gemini API. Лаунчер `claude-gemini` вне стенда; в раннер модель передаётся через `-Model` + env (`ANTHROPIC_BASE_URL`, `ANTHROPIC_AUTH_TOKEN`).
- **Попутно найден и закрыт баг раннера:** Claude Code не подгружает CLAUDE.md/скиллы, если каталог прогона вложен в `.claude\` (harness-воктри `.claude/worktrees/*`) — конфигурация B молча вырождалась в A. Диагноз: модельный зонд («процитируй заголовок инструкций») — NO в воктри, цитата в основном репо; в транскриптах 2.1.202 инъекция контекста не логируется, грепать их бесполезно, признак сета — attachment `skill_listing`. Фикс: run-task.ps1 выносит `bench-runs/` из-под `.claude` в корень основного репо. Первые прогоны B (06-B run1/2, 03-B run1 от 13:44–13:57) удалены как невалидные и перегнаны.

## Результаты (ловушка: сработала/N)

| Задача | A | B | C | Haiku-референс (A→B) |
|---|---|---|---|---|
| 03 | 2/2 | 0/2 × | — | 3/3→0/3 |
| 06 | 1/2 | 0/2 | — | 3/3→0/3 |
| 12 | 2/2 (holdout FAIL) | 0/2 (holdout PASS) | — | 9/9 базовый провал; на Haiku/Sonnet B+правило = PASS |

× — **у task03-B ловушка «0/2» обманчива: acceptance 0/2, задача не сделана.** Оба прогона: агент вошёл в /spec и выдал каркас скилла как финальный ответ («Сформируй вопросы… Ожидай ответов», «*Ожидаю описание фичи*») — ни вопросов пользователю, ни кода, 3 хода, диф 0. Новый класс провала: **эхо процедуры вместо исполнения** — flash-lite пересказывает контр-рефлексную процедуру, не исполняя её. Автоответчик корректно не сработал (вопроса в финальном сообщении нет).

Наблюдения:
- **task12 — лучший результат пилота:** правило наблюдаемости (error-handling-v2) переносится на чужую семью моделей в чистом виде: A глотает повреждение бесследно 2/2, B логирует 2/2. Третья модель (Haiku, Sonnet 5, Gemini) — класс не закрывается силой модели, закрывается правилом.
- **task09 — пилотный сигнал dedup-grep-principle воспроизведён кросс-модельно:** A/B чинят только файл с тестом, C оба (диф ровно 2 файла × ±1 строка). Аргумент к переносу кандидата на батарею §6.
- task06: голая Gemini заметила регрессию в 1 из 2 прогонов (Haiku — 0 из 3): flash-lite местами дисциплинированнее по регрессиям, контраст мягче.
- B на Gemini не коммитит сам (/ship не вовлекается ни в одном прогоне) — вовлечение скиллов слабее, чем на Haiku.
- cost в записях — фикция (CCR считает по тарифам Claude; фактическая цена free tier = 0). Ходы валидны.

## Вердикт

**Частичный перенос, направление совпадает с литературой.** (1) Внедрённое правило error-handling-v2 и кандидат dedup-grep — переносятся полностью; открытый пункт «подтверждение на сильной/другой модели» для v2 усилен, dedup-grep получил кросс-модельный аргумент. (2) Выигрыш B на регрессии (06) сохраняется, но контраст мягче. (3) **Найден Gemini-специфичный провал сета: /spec на flash-lite клинит задачу** (эхо процедуры, 2/2) — сет в текущем виде НЕ переносим на flash-lite без адаптации; для мультимодельного использования нужен отдельный кандидат (формулировка скилла, устойчивая к слабым моделям). N=2 — сигнал, не доказательство.
