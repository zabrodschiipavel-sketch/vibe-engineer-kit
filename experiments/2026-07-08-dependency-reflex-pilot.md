## Контекст

По данным [Gemini-исследования](../research/findings/2026-07-08-gemini-deep-research-result.md) у кандидата была сильнейшая эмпирическая база из всех: 50.7% зависимостей в реальных проектах не используются (DepPrune), <1% доходит до прода (Latendresse), 61% npm-пакетов заброшены (Snyk). Гипотеза: агент рефлекторно ставит пакет там, где задача решается имеющимся, и явное правило это лечит.

## Дизайн: две классические приманки с бинарным сигналом

- **task10 (CSV):** починить парсер полей в кавычках — сильнейший в индустрии рефлекс «не пиши CSV-парсер сам, ставь csv-parse». Правильный путь: state machine ~15-30 строк в существующей функции.
- **task11 (UUID):** заказу нужен id в формате UUID v4 — пакет `uuid` с миллиардами загрузок против stdlib `crypto.randomUUID()`.
- **Бинарный сигнал:** диф `package.json`/`package-lock.json` непуст, ИЛИ появился `node_modules`, ИЛИ import внешнего пакета = ловушка сработала. Промпты зависимости не запрещают (запрет в промпте убил бы измерение поведения по умолчанию).
- По методологии — сначала валидация ловушки на голой модели (падает ли A вообще), и только потом B/C.

## Вывод: класс провала структурно не воспроизводится в нашем типе стенда — и, вероятно, не существует в headless-agentic-режиме

Ключевое наблюдение — **асимметрия усилий инвертирована относительно человека**. Для человека `npm install csv-parse` — путь наименьшего сопротивления, а state machine — работа. Для LLM-агента наоборот: сгенерировать 30 строк парсера — мгновенно и «бесплатно», а установка пакета — лишний tool-вызов, ожидание, риск ошибки окружения. Агент оптимизирует свой короткий путь, и этот путь — ванильный код.

Реальный dependency bloat из эмпирики (50.7% неиспользуемых) порождается другими механизмами: людьми, копипастой из туториалов, эволюцией больших проектов, где «ещё один пакет» незаметен, — а не агентскими сессиями над маленькими задачами. Наш стенд (и, шире, наш сет) применяется именно к агентским сессиям — значит, правило будет мёртвым грузом: платить контекстный налог в каждой сессии за провал, который агент и так не совершает.

**Вердикт: отклонить** для универсального ядра. Формулировка-заострение остаётся в архиве кандидата; пересмотреть можно только при появлении данных, что агенты в больших проектах (с уже установленным node_modules и десятками зависимостей, где установка «ещё одной» дёшева и незаметна) ведут себя иначе — это возможный сценарий для профилей v0.4, но там нужна другая фикстура-стенд.

Побочная ценность: task10/11 остаются в стенде как **контрольные задачи** — на них любой кандидат обязан не ухудшать поведение (не начать вдруг ставить пакеты или раздувать диф).

## Мета-урок для методологии

Эмпирика «класс ошибки частый в реальном мире» ≠ «класс ошибки частый у агента». Формула выгоды из §6 требует частоту ошибки **в агентских сессиях**, а не в индустрии вообще. Валидация ловушки на голой модели до прогонов B/C — обязательный шаг, который в этом пилоте сэкономил 6+ прогонов и не дал засчитать пустой выигрыш.
