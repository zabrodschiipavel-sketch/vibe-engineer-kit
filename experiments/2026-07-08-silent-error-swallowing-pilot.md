## Дизайн

Две приманки, где промпт требует «не падать», а короткий путь — молчаливый `try/catch`:
- **task12:** битый файл настроек — повреждение (потеря пользовательских данных) становится неотличимым от отсутствия файла (первый запуск);
- **task13:** сбой одного получателя рассылки — частичный отказ становится неотличимым от полного успеха.

Бинарный сигнал (скрытые holdout'ы): **наблюдаемость** сбоя — есть ли у вызывающего кода/оператора хоть один канал узнать о проблеме (лог, маркер в результате, итоговое исключение).

## Два независимых вывода

### 1. Класс провала у агентов РЕАЛЕН (в отличие от dependency-reflex)

task12 ловит 9/9 — все конфигурации на всех прогонах пишут `catch { return DEFAULTS }`, делая порчу пользовательских настроек бесследной. Это подтверждает агентскую частоту класса «graceful degradation съедает контекст ошибки». task12 остаётся в стенде как **сильнейшая дифференцирующая ловушка** (максимальный контраст: базовый провал 100%).

task13 дифференцирует слабо (голая модель в 2/3 случаев сама делает сбой наблюдаемым — итоговым исключением) — оставлена как вторичная.

### 2. Формулировка-кандидат ИНЕРТНА: 0 эффекта при явном правиле в CLAUDE.md

C проглотил ошибку так же, как A и B: 3/3 на task12 — буквально написав `catch { return DEFAULTS; }` без лога и без обоснования, которые правило прямо требует. Ни один C-прогон даже словами не упомянул компромисс.

**Почему (гипотеза, согласуется со всеми данными):** промпт задачи легитимизирует поведение («при проблемах продолжаем на дефолтных») — и конкретная инструкция задачи доминирует над общим нормативным правилом из CLAUDE.md. Контраст с успешным пилотом dedup-grep-principle (3/3 эффекта) наводит на паттерн:

> **Процедурные правила («после X сделай действие Y») меняют поведение агента; нормативные правила («не делай плохо Z») перекрываются рамкой задачи.**

Это гипотеза по N=2 кандидатам — не факт. Но она напрямую проверяема: переписать кандидата в процедурной форме («после добавления try/catch вокруг I/O — добавь в catch лог или комментарий-обоснование») и прогнать на той же task12 с её 100%-контрастом. Это отдельный будущий кандидат (`error-handling-procedural`), не правка текущего.

## Вердикт: отклонить текущую формулировку

По протоколу: C = B = A, прирост нулевой — не «доисследовать» (данные не шумные, контраст задачи 100%, эффект отсутствует чисто). Нормативная формулировка отклонена; производная процедурная гипотеза добавлена в индекс кандидатов.
